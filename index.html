cooking and modeling processes

Following is a set of concepts needed for modeling processes with analogies to cooking. This is a rough, informal sketch of ideas to be fleshed out more formally.

  • resources / ingredients
  • timers / waiting
  • verbs / operations from known vocabulary: chop / mix etc
  • perishable / expiring resources
  • constraints: don't leave empty pan on fire for more than N seconds
  • alternate context for terms within scope of recipe: substitute vegan versions of all animal-based ingredients
  • running out of something, temporarily, requesting / waiting for replenishment / service
  • oven simulation using force layout (opening door bumps it towards room temp, which could itself be a continuous variable simulation)
  • event: when oven reaches 350°, put prepared pan into oven
  • referencing things / naming composites. "the prepared pan", "the stuffing"
  • attribute queries: "the wet ingredients" and "the dry ingredients"
  • irreversible transforms: aggregates, e.g. mixing liquids
  • output of one process is input to another
  • state machine: when a pancake begins to bubble, flip it. then when it no longer sticks to the pan, remove it from pan (dequeue)
  • event from stream with threshold condition: when the onions are browned, add the broccoli
  • error handling: if you burn what's in the pan, throw it out and restart that part of the process over
  • precondition: ensure knives are sharpened
  • inference rule: range/domain. chopping is only for solids, pouring is only for liquids
  • event from external system: rice cooker switches to "finished"
  • synchronization: (when the crust and the filling are complete), put the filling into the crust
  • parallelization: while the oven is preheating / while the dough is rising
  • parallel with constraints: use one of N burners
  • transforms with possible reverse mapping: map, filter (cookie dough ball <-> cookie)
  • instance of another model: serve with the bread from recipe X
  • Use output of another process as input to this process: with the frosting from recipe X, frost the finished cake
  • multiple instances of another model: make two of the flaky pie crusts
  • arbitrary selection: make a pie crust of your choice (or 2)
  • parameterization: multiply recipe, use variant (whole-wheat)
  • rules: 2 cups water to one cup white rice
  • rule specializations: 2.5 cups water for brown rice
  • override: substitute ingredients
  • blocking queues: fixed-capacity cookware (e.g boil 6 eggs at a time, dequeue when it floats)
  • environmental factors: humidity, air pressure and temperature.
  • batching
  • loops: repeat (sub) process for each stuffed pepper
  • environmental factors: choose a seasonal/regional vegetable
  • state: make a cookie with one spoonful of batter until you run out. generally, things in any state of preparation. state of stove range, N burners
  • dataflow: room / oven temperature. "cookedness" of something as a continuous (discretized) variable. water flow filling container. ratio of size of dough rising (e.g. when the dough has doubled in size)
  • external events: timer, arrival of delivery (e.g. for ingredients)
  • reflected (not reified) processes. e.g. consumers taking from queue (i.e. people walking into the kitchen and taking from stack as you cook). simulation, can you reach a threshold amount with different rates of production / consumption
  • units (tbsp, tsp, cup, quart)

Enumerated sets

A set is a kind of collection

RDF includes a notion of containers but not of sets specifically. However, RDF resources are in some ways sets by default, in that graphs are sets of triples, and not multisets.

See As Shelley Powers notes in Chapter 4 from Practical RDF,

use the RDF collection as sparingly as you would use the RDF Container—use only when no other construct matches your specific needs, and use it specifically as the specification intended it to be used. If you’re unsure about the intent, then don’t use it.

Dropping into this collection might cause an rdf:member assertion to be made about the item being dropped. This could be implied by an RDFa rel attribute on the container.

Duplicate items should not be accepted (but currently are).

In mathematics, sets are usually denoted with curly braces {}

HTML doesn't have any notion about sets as such. The closest thing is the unordered list, which can be used to represent an enumerated set. However, it doesn't have a notion of uniqueness.

MELD preamble

This is (a working draft of) the MELD preamble.

Background

MELD is a project for creating a dynamic medium on top of HTML documents. A MELD document is anything you can think of: an interactive presentation, a model of the Earth, a personal computing environment. MELD supports the arbitrary organization of “programs” into documents and is itself is embodied in HTML fragments, meaning that you could tuck it away in an appendix or scatter its parts as you see fit. However, because of the way that HTML works, some of that foundation must physically (i.e. lexically) precede everything else in order for the system to work. That's what this preamble is for: to provide a minimal basis for everything else.

(This still does not precede the document head. MELD systems avoid using the document head because it's invisible.)

Built-in capabilities : what we get “for free”

In this section, we look at (yes, look at) the web platform's built-in capabilities supporting dynamic documents. While they are not sufficient for all of our needs, we can use these features with minimal effort and build on them.

By “free” I mean “without script.” If we have to implement it ourselves, it's not free.

Extending the built-in capabilities

defensiveness

The objective of portability is that (to the extent possible) a document can travel from one environment to another and retain its essential character (for lack of a better word).

Some of that preservation relies on those capabilities that will become unavailable in standalone mode. Therefore, a portable document must pre-emptively take the steps that will allow it to survive in the wild.

What this means in practice is that we need to inline scripts (and other things that come from the host) as soon as the document is loaded.

This script finds and inlines all of the remote scripts in the document by fetching their content. Since it fetches an arbitrary resource from an arbitrary location, it will only work in hosted mode. House rules

The following style rules are included just to make the document more readable. They are not essential to the functionality being discussed. This does, however, illustrate a usage of dynamic documents.

Following are common rules to make narrative document content more readable than browser defaults.

The aside element, although it has to be placed at some definite point in the document, is not part of the document “flow.”

Blockquotes traditionally have certain formatting. Some of that, such as the left margin, is a default style in most browsers.

I'm a fan of the “disclosure” elements. They provide a free way to hide and show things.

The summary element is affordable (you can act on it), but the default styles do not indicate this. 

              summary:hover {
                cursor: pointer;
                background-color: rgba(0, 0, 0, 0.3);
              }
This is now covered by the element stylesheet.

I'm using disclosure elements to make sections collapsible. For some reason (I guess related to built-in rules), block-level elements do not appear when used inside of a summary. This fixes that. 

              summary > h2,
              summary > h3,
              summary > h4 {
                display: inline;
              }
But this has unwanted effects when used with element visibility rules, which prevent the problem noted above.

By default, text inputs are far too small. If I wanted something that small, I'd use an input. A textarea should be like a block-level element, which automatically uses the available horizontal space.

In general, we would prefer that scripts be editable, in which case they would use a textarea. However, we do still see some script elements, which are read-only. Scripts need to be interpreted differently than their surrounding content, and thus should be demarked in some way, to cue the change in context.

But how should scripts be demarked? Many factors bear on this. I will not go into all of them now, but in general, I believe that physical demarcations should be reserved to indicate physical distinctions. While I personally prefer proportional fonts for code, I would note that the JavaScript code in question here is not what we ultimately want to be reading anyway.
